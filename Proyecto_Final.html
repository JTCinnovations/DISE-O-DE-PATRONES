1) Objetivo

Construir un sistema en Java que resuelva un problema real y demuestre dominio de SOLID, Patrones GoF y GRASP, con arquitectura clara, pruebas y documentación.

Java 17+ Plataforma libre Evidencias de diseño

2) Alcance mínimo (requisitos funcionales)

  • Un flujo de negocio completo (inicio → procesamiento → salida/reportes).
  • Gestión de entidades (crear/leer/actualizar/borrar, al menos desde la lógica).
  • Persistencia con repositorios claros (BD relacional/NoSQL/archivos/embebida).
  • Interfaz a elección: Escritorio (JavaFX), CLI (picocli), Android, Microservicios, IoT, Batch/ETL.
Consejo: privilegia un alcance realista y bien probado sobre un sistema enorme sin evidencias de diseño.

3) Requisitos de diseño

3.1 SOLID (obligatorio: los cinco)

  • SRP: una sola razón de cambio por clase.
  • OCP: extensión por nuevas clases/estrategias, sin modificar el núcleo.
  • LSP: subtipos sustituibles por su supertipo sin romper contratos.
  • ISP: interfaces pequeñas, específicas por cliente.
  • DIP: depender de abstracciones; usar inyección de dependencias.

3.2 Patrones GoF (mínimos)

CategoríaMínimoEjemplos
Creacionales≥ 2Factory Method, Abstract Factory, Builder, Prototype, Singleton (evitar abuso)
Estructurales≥ 2Adapter, Facade, Composite, Decorator, Proxy, Bridge
Comportamiento≥ 2Strategy, Observer, Command, State, Template Method, Chain of Responsibility, Memento

3.3 GRASP (≥2)

Recomendados: Controller, Information Expert, Creator, Low Coupling, High Cohesion, Polymorphism, Pure Fabrication.

Evita antipatrones: God Class, Feature Envy, exceso de condicionales anidados, abuso de Singleton, duplicación de lógica.

5) Calidad y pruebas

  • Pruebas unitarias (JUnit 5 + Mockito) y al menos 1–2 de integración (H2/SQLite o Testcontainers).
  • Cobertura significativa del flujo principal y de ≥1 patrón por categoría.
  • Manejo de errores y logging (INFO/ERROR); datos semilla.

6) Entregables

  1. Código con README ejecutable.
  2. Documentación técnica (PDF/MD): arquitectura, tablas de evidencias (SOLID/GoF/GRASP), capturas o registros.
  3. Pruebas (cómo correrlas y breve reporte).
  4. Video demo (5–8 min).
  5. Presentación (8–12 diapositivas).

9) Ideas de proyectos (no web) para inspirar

  • Gestor de laboratorio (JavaFX): inventario, reservas, alertas (Observer), perfiles (Strategy), importación CSV (Adapter), reportes (Facade).
  • Planificador de rutas offline (CLI): grafos, heurísticas (Strategy/Template), exportes (Adapter), “deshacer” (Memento/Command).
  • Monitor de producción (IoT): telemetría (Observer), drivers (Bridge), familias de sensores (Abstract Factory), restauración de config (Memento).
  • Procesador ETL de facturas (Batch): pipeline (Template Method), validaciones (Chain of Responsibility), conectores (Adapter).

11) Conceptos avanzados sugeridos (opcionales para subir nota)

  • Reflexión en Java para descubrimiento dinámico de plugins (combinar con Abstract Factory + Strategy). 
    // Descubrir estrategias por convención de paquete
ServiceLoader<PricingStrategy> loader = ServiceLoader.load(PricingStrategy.class);
for (PricingStrategy s : loader) {
    System.out.println("Estrategia detectada: " + s.getClass().getSimpleName());
}
  • Anotaciones personalizadas para validación declarativa (mini framework tipo Bean Validation) + Template Method para el pipeline de validación. 
    @Target(ElementType.FIELD) @Retention(RetentionPolicy.RUNTIME)
public @interface NotEmpty {}

public abstract class ValidatorTemplate<T> {
  public final void validate(T t) { pre(); doValidate(t); post(); }
  protected void pre(){} protected void post(){}
  protected abstract void doValidate(T t);
}
  • Inyección de dependencias ligera (DIP) construida por el equipo (mini contenedor) para reforzar OCP y probar dobles de test. 
    public class Container {
  private final Map<Class<?>, Object> instances = new HashMap<>();
  public <T> void bind(Class<T> type, T instance){ instances.put(type, instance); }
  public <T> T get(Class<T> type){ return type.cast(instances.get(type)); }
}
  • Event-Driven con Observer y Domain Events (asincronía simulada con colas en memoria).
  • Patrones arquitectónicos (Hexagonal/Onion/Clean) y DTO/Mapper para aislar capas.
  • Medición de calidad (métricas de complejidad, Sonar, Jacoco) y técnicas de refactor (Extract Class/Method, Replace Conditional with Polymorphism).
Tip: documenta cómo los conceptos avanzados refuerzan SOLID y reducen acoplamiento.
